# Task 3: Collection Definitions

## Design Overview

This database uses a single collection called `clubs` where everything is nested inside. Instead of splitting data across multiple tables like in a relational database, all the related information (members, events, RSVPs) is stored together in one place.

The main reason for this design is that when you look up a club, you usually want to see its events and who's attending them all at once. By keeping everything together, MongoDB can grab all that data in one go instead of having to connect multiple tables.

The downside is that student information gets repeated if they're in multiple clubs, but that's fine - storage is cheap and this way is way faster for reading data.

**Indexes to add:**
- club_id (make it unique)
- events.event_id
- events.submission.status
- members.student_id

## Why This Design?

**Single collection with everything embedded:**

The main idea is that when you look up a club, you almost always want to see what events they're hosting and who's going to them. By keeping everything in one document, MongoDB can grab it all in one database call instead of joining multiple tables together.

**Data gets repeated:**

Yes, if Alice is in 3 different clubs, her information (name, email, major) appears 3 times in the database. In a traditional SQL database, you'd avoid this by having a separate Students table. But in MongoDB, this repetition is fine because:
- Hard drives are cheap
- Reading is way faster this way
- Student info doesn't change that often anyway

**When to use this structure:**

- Getting all clubs: `db.clubs.find()`
- Getting one club with everything: `db.clubs.findOne({club_id: 1})`
- Finding all academic clubs: `db.clubs.find({category: "academic"})`
- Finding what events a student RSVPed to: `db.clubs.find({"events.rsvps.student.student_id": 1})`
- Counting total RSVPs across all events: Use aggregation with $unwind
