Design Notes (for the engineer taking over this database)
This database supports a College Event Planning system. The schema uses a hybrid embedding strategy:
Small, bounded subdocuments (officers, RSVPs, submissions) are embedded to avoid unnecessary joins and because they are always accessed with their parent.
Cross-collection relationships use student_id, club_id, etc. as manual references (not DBRefs), which is standard for MongoDB at this scale.
Recommended indexes: student_id on Student, club_id on Event, email fields across all collections for lookup performance.

# University Club Management System: Upgrading to a MongoDB Database

## What is this project all about?
This project is a complete redesign of a university club tracking system. Originally, this system was built using a traditional, old-school database (SQL). In that older version, information was scattered across many different, separate tables. For example, to find out which club a student was in, the database had to constantly play connect-the-dots between a "Student" table, a "Membership" table, and a "Club" table. 

The goal of this project was to upgrade that system to a modern, document-based database called **MongoDB**. 

Instead of scattering the data, a document database allows us to store related information together in one place. By grouping the data logically—like putting a student's club memberships directly inside their personal file—we make the database much faster and easier for the computer to read. 

## How We Organized the Data
We took 8 separate lists of information and reorganized them into 4 main "Folders" or Collections:

1. **Clubs:** This holds the main details of the club (like its name and category) and also directly lists the officers who run it. 
2. **Students:** This contains the student's personal information (major, graduation year) and a list of all the clubs they are a member of.
3. **Events:** This is the most detailed file. It holds the event details, the administrative approval status for that event, and a full list of every student who has RSVP'd to attend.
4. **Admins:** A separate list for the university staff who are responsible for approving or rejecting event requests.

## What is Included in this Repository
* **The Blueprint:** We have included the exact templates (called JSON schemas) that show how our new data files are structured and organized. 
* **Sample Data:** A database isn't very helpful if it's empty! We used a tool called Mockaroo to generate hundreds of realistic, fake students, clubs, and events to fully test our new system. The files to load this data are included in the `/data` folder.
* **Database Commands (Queries):** We wrote out specific instructions to prove the new database works. These include commands to calculate total event capacities, find specific types of clubs, count how many events a single student is attending, and update a student's RSVP status.
* **A Mini Web Application:** We built a basic web server using Node.js. This allows a user to actually interact with our database from a web browser to view, add, update, or delete information.

## Tools and Technology Used
* **MongoDB & MongoDB Compass:** The main software used to build and view our new document database.
* **Node.js & Express.js:** The programming languages used to build the backend web server.
* **Mockaroo:** A website used to generate the realistic test data.
* **LucidChart:** Used to draw the original maps and diagrams of the system.
